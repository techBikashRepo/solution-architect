### 1. What is it?

**Amazon S3** is AWS's object storage service — store and retrieve any amount of data, from anywhere, at virtually unlimited scale. Objects (files) are stored in **buckets**, accessed via URL or API.

Think of S3 as: an infinitely large hard drive accessible over HTTPS with 11 nines (99.999999999%) durability.

---

### 2. Key Concepts

| Concept           | Detail                                                        |
| ----------------- | ------------------------------------------------------------- |
| **Bucket**        | Container for objects; globally unique name; tied to a region |
| **Object**        | A file + metadata; max 5TB per object                         |
| **Key**           | Object's "path" within bucket: `images/2025/photo.jpg`        |
| **Version ID**    | Unique ID per version when versioning enabled                 |
| **Prefix**        | Simulated folder path (S3 has no real folders)                |
| **Storage Class** | Tier determining cost and retrieval time                      |
| **Presigned URL** | Temporary signed URL granting time-limited access             |
| **Bucket Policy** | JSON policy controlling access to the entire bucket           |
| **ACL**           | Legacy per-object access control (avoid; use bucket policies) |
| **CORS**          | Configure cross-origin access for browser uploads             |

---

### 3. Storage Classes (Cost Optimization — Critical)

| Class                             | Use Case                                       | Min Duration | Retrieval     | Relative Cost              |
| --------------------------------- | ---------------------------------------------- | ------------ | ------------- | -------------------------- |
| **S3 Standard**                   | Frequently accessed data                       | None         | Instant       | Baseline                   |
| **S3 Intelligent-Tiering**        | Unknown access patterns                        | None         | Instant       | ~Baseline (auto-optimizes) |
| **S3 Standard-IA**                | Infrequently accessed, instant needed          | 30 days      | Instant       | ~40% cheaper               |
| **S3 One Zone-IA**                | Non-critical, IA, one AZ only                  | 30 days      | Instant       | ~55% cheaper               |
| **S3 Glacier Instant Retrieval**  | Archives needing <1ms retrieval                | 90 days      | Milliseconds  | ~68% cheaper               |
| **S3 Glacier Flexible Retrieval** | Archives; expedited (1-5min), standard (3-5hr) | 90 days      | Minutes/Hours | ~80% cheaper               |
| **S3 Glacier Deep Archive**       | Rarely accessed; compliance                    | 180 days     | 12-48 hours   | ~95% cheaper               |

---

### 4. S3 Features for System Design

| Feature                   | How It Works                                                        |
| ------------------------- | ------------------------------------------------------------------- |
| **Versioning**            | Keep all versions of every object; protect from overwrites/deletes  |
| **Lifecycle Rules**       | Auto-transition to cheaper class or auto-delete after N days        |
| **Replication (CRR/SRR)** | Auto-replicate objects to another region (CRR) or same region (SRR) |
| **Event Notifications**   | Trigger Lambda/SQS/SNS on object operations (PUT, DELETE)           |
| **S3 Select**             | Query inside objects using SQL (query CSV/JSON without downloading) |
| **Transfer Acceleration** | Route uploads through CloudFront edge → faster for distant users    |
| **Multipart Upload**      | Upload large objects in parts (required >5GB, recommended >100MB)   |
| **Object Lock**           | WORM (Write Once Read Many) — prevent deletion for compliance       |

### 5. S3 Access Patterns

```
PRESIGNED URL (most common for user-facing file operations):
─────────────────────────────────────────────────────────
  # Server generates presigned URL:
  import boto3
  s3 = boto3.client('s3')
  url = s3.generate_presigned_url(
      'get_object',
      Params={'Bucket': 'my-bucket', 'Key': 'file.pdf'},
      ExpiresIn=3600  # 1 hour
  )
  # Returns: https://my-bucket.s3.amazonaws.com/file.pdf?X-Amz-Signature=...
  # Client uses this URL directly; no AWS credentials needed

BUCKET POLICY (public-read for static website):
  {
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-website-bucket/*"
    }]
  }

LIFECYCLE RULE (cost tiering):
  Transition: after 30 days → Standard-IA
  Transition: after 90 days → Glacier Instant Retrieval
  Expiration: after 365 days → delete
```

### 6. When to Use S3

✅ **Use S3 for**:

- Static website hosting (S3 + CloudFront)
- Media storage (images, videos, documents)
- Data lake (raw data → Athena for querying)
- Backup and disaster recovery
- Application deployment artifacts
- CloudTrail / CloudWatch logs archival
- ML training datasets

❌ **Don't use S3 for**:

- Low-latency block storage → use EBS
- Shared file system across EC2 → use EFS
- Database storage → use RDS/DynamoDB
- Frequent small object reads (ListObjects at scale is costly)

### 7. S3 Security Best Practices

```
BLOCK PUBLIC ACCESS (always enable on all buckets):
  s3:BlockPublicAcls: true
  s3:IgnorePublicAcls: true
  s3:BlockPublicPolicy: true
  s3:RestrictPublicBuckets: true

  Exception: buckets hosting public static websites

ENCRYPTION:
  Server-Side Encryption (SSE-S3):    AWS manages keys (free, default since 2023)
  Server-Side Encryption (SSE-KMS):   Customer manages keys in KMS (audit trail)
  Client-Side Encryption:             Encrypt before uploading (maximum control)

BUCKET POLICY vs IAM:
  IAM Policy: controls what an IAM principal can do
  Bucket Policy: controls what can be done to the bucket (including cross-account)

  Rule: use IAM for same-account access; use Bucket Policy for cross-account and public

VERSIONING + OBJECT LOCK for compliance:
  Enable MFA Delete: require MFA to delete objects or disable versioning
  Object Lock (WORM): legal holds, compliance mode (can't be deleted ever), governance mode
```

### 8. AWS Architecture Examples

```
STATIC WEBSITE WITH CDN:
  [S3 bucket: www.myapp.com] (static HTML/CSS/JS)
  [CloudFront distribution]
    Origin: S3 bucket
    SSL: ACM certificate
    Cache: TTL 24hr for assets, 5min for HTML
  [Route 53] → ALIAS record → CloudFront

  Result: global CDN delivery; sub-second page loads worldwide

DATA LAKE:
  [Applications] → write raw data → [S3: raw/]
  [ETL Lambda/Glue] → transforms → [S3: processed/]
  [Athena] → SQL queries on S3 data (pay per query)
  [QuickSight] → dashboards from Athena

  S3 Lifecycle: raw/ → 90 days → Glacier; processed/ → 30 days → IA

FILE UPLOAD PIPELINE:
  [Client] ← presigned URL ← [App Server]
  [Client] → PUT file → [S3: uploads/]
  [S3] → event notification → [Lambda: process-file]
  [Lambda] → read S3 → generate thumbnail → write [S3: thumbnails/]
  [Lambda] → update [DynamoDB]: file status=available

CROSS-REGION REPLICATION (DR):
  [S3: us-east-1] → CRR → [S3: us-west-2]
  Enable versioning on both buckets (required)
  RPO: seconds (replication near real-time)
  For compliance: keep replica in different region
```

### 10. Common Interview Questions

**Q1: What is the difference between S3 Standard and S3 Intelligent-Tiering?**

> S3 Standard: flat pricing regardless of access patterns. Best when data is frequently accessed. S3 Intelligent-Tiering: automatically moves objects between access tiers (frequent → infrequent → archive) based on access patterns. Zero retrieval fees. Small monitoring fee per object (~$0.0025 per 1,000 objects/month). Best for data with unknown or changing access patterns. Rule of thumb: for > 128KB objects with unpredictable access, Intelligent-Tiering often saves money over Standard. For small objects (<128KB) or guaranteed frequent access, Standard is simpler and cheaper.

**Q2: How would you set up S3 for HIPAA compliance?**

> Requirements: encryption, access logging, audit trails, no public access. Implementation: (1) Enable Block Public Access on all buckets. (2) Enable SSE-KMS encryption (provides CloudTrail audit trail of key usage). (3) Enable S3 Server Access Logging → separate audit bucket. (4) Enable CloudTrail for S3 data events (track who accessed what, when). (5) Enable Versioning + Object Lock (WORM) for medical records that must not be modified. (6) Enable S3 Replication to a second region for DR (healthcare data retention requirements). (7) VPC Endpoint for S3: ensure traffic never leaves AWS network.

**Q3: How would you handle 10,000 concurrent S3 uploads?**

> S3 scales automatically — no configuration needed for throughput. However, performance tip: S3 achieves at least 3,500 PUT and 5,500 GET requests per second per prefix. If your uploads all use the same prefix (e.g., `uploads/2025-01-15/`), you hit the per-prefix limit with 3,500+ concurrent uploads. Solution: randomize prefixes: `uploads/{hash_prefix}/{filename}`. The hash prefix spreads objects across S3 partitions. For very large files: multipart upload (required for >5GB, recommended for >100MB) — split into 5-25MB parts, upload in parallel, S3 reassembles.
